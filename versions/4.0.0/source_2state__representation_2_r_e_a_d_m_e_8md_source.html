1 # State Representation
2 
3 This library provides a set of classes to represent **states** in **Cartesian** or **joint** spaces, **parameters**,
4 or **geometrical shapes** that can be used as obstacles.
5 Those are a set of helpers functions to handle common concepts in robotics such as transformations between frames and
6 the link between them and the robot state.
7 This description covers most of the functionalities starting from the spatial transformations.

24 ## Cartesian state
25 
26 A `CartesianState` represents the transformations between frames in space as well as their dynamic properties
27 (velocities, accelerations and forces).
28 It comprises the name of the frame it is associated to and is expressed in a reference frame (by default `world`).
29 A state contains all the variables that define its dynamic properties, i.e `position`, `orientation`, `linear_velocity`,
30 `angular_velocity`, `linear_acceleration`, `angular_acceleration`, `force` and `torque`.
31 All those state variables use `Eigen::Vector3d` internally, except for the orientation that is `Eigen::Quaterniond` based.
32 All getters and setters are implemented.
33 
34 ```cpp
35 state_representation::CartesianState s1("a"); // frame a expressed in world (default)
36 state_representation::CartesianState s2("b", "a"); // frame b expressed in a
37 
38 s1.set_position(Eigen::Vector3d(0, 1, 0)); // 1 meter in y direction
39 s1.set_orientation(Eigen::Quaterniond(0, 1, 0, 0));
40 ```
41 
42 By default, quaternions are normalized on setting, therefore:
43 
44 ```cpp
45 // will be rendered as Eigen::Quaterniond(0.70710678, 0.70710678, 0. , 0.)
46 s2.set_orientation(Eigen::Quaterniond(1, 1, 0, 0));
47 ```
48 
49 The state variables are also grouped in `pose` (`position` and `orientation`), `twist` (`linear_velocity` and
50 `angular_velocity`), `accelerations` (`linear_acceleration` and `angular_acceleration`) and `wrench` (`force` and `torque`).
51 Getter and setters are also implemented to do those bulk operations.
52 
53 ```cpp
54 s2.set_twist(Eigen::VectorXd::Random(6));
55 ```
56 
57 Note that for `pose`, it will be a `7d` vector (3 for `position` and 4 for `orientation`):
58 
59 ```cpp
60 s2.set_pose(Eigen::VectorXd::Random(7));
61 ```

63 ### Cartesian state operations
64 
65 Basic operations between frames such as addition, subtractions, scaling are defined, and applied on all the state variables.
66 It is very important to note that those operations are only valid if both states are expressed in the same reference frame.
67 
68 ```cpp
69 state_representation::CartesianState s1("a"); // reference frame is world by default
70 state_representation::CartesianState s2("b");
71 double lamda = 0.5
72 
73 // for those operation to be valid both s1 and s2
74 // should be expressed in the same reference frame
75 state_representation::CartesianState ssum = s1 + s2;
76 state_representation::CartesianState sdiff = s1 - s2;
77 state_representation::CartesianState sscaled = lambda * s1;
78 ```
79 
80 Also, despite common mathematical interpretation, the `+` operator is **not commutative** due to the orientation part of the state.
81 The addition `s1 + s2` corresponds to the transformation from the `world` frame to frame `a` followed by the transformation
82 from `world` frame to frame `b`.
83 If both frames have the same orientation then the `+` operator is commutative.
84 
85 ```cpp
86 state_representation::CartesianState s1("a");
87 state_representation::CartesianState s2("b");
88 s1.set_orientation(Eigen::Quaterniond(0, 1, 0, 0));
89 s2.set_orientation(Eigen::Quaterniond::UnitRandom());
90 
91 state_representation::CartesianState ssum1 = s1 + s2;
92 state_representation::CartesianState ssum2 = s2 + s1;
93 
94 ssum1 != ssum2;
95 ```

97 ### Changing of reference frame
98 
99 One of the most useful operation is the multiplication between two states that corresponds to a changing of reference frame:
100 
101 ```cpp
102 state_representation::CartesianState wSa("a"); // reference frame is world by default
103 state_representation::CartesianState aSb("b", "a");
104 
105 // for this operation to be valid aSb should be expressed in a (wSa)
106 // the result is b expressed in world
107 state_representation::CartesianState wSb = wSa * aSb;
108 ```
109 
110 Not only does that apply a changing of reference frame but it also express all the state variables of `aSb`
111 in the desired reference frame (here `world`), taking into account the dynamic of the frame `wSa`,
112 i.e. if `wSa` has a `twist` or `acceleration` it will affect the state variables of `wSb`.
113 
114 ### Specific state variables
115 
116 Full `CartesianState` can be difficult to handle as they contain all the dynamics of the frame when, sometime,
117 you just want to express a `pose` without `twist`, `accelerations` or `wrench`.
118 Therefore, extra classes representing only those specific state variables have been defined, `CartesianPose`,
119 `CartesianTwist` and `CartesianWrench`.
120 Effectively, they all extend from `CartesianState` hence they can be intertwined as will.
121 
122 ```cpp
123 state_representation::CartesianPose wPa("a");
124 state_representation::CartesianState aSb("b", "a");
125 
126 // the result is state b expressed in world
127 state_representation::CartesianState wSa = wPa + aSb;
128 ```
129 
130 ```cpp
131 state_representation::CartesianPose wPa("a");
132 state_representation::CartesianTwist aVb("b", "a");
133 
134 // the result is twist b expressed in world
135 state_representation::CartesianTwist wVa = wPa + aVb;
136 ```

138 ### Conversion between Cartesian state variables
139 
140 The distinction with those specific extra variables allows to define some extra conversion operations.
141 Therefore, dividing a `CartesianPose` by a time (`std::chrono_literals`) returns a `CartesianTwist`:
142 
143 ```cpp
144 using namespace std::chrono_literals;
145 auto period = 1h;
146 
147 state_representation::CartesianPose wPa("a", Eigen::Vector3d(1, 0, 0));
148 // the result is a twist of 1m/h in x direction converted in m/s
149 state_representation::CartesianTwist wVa = wPa / period;
150 ```
151 
152 Conversely, multiplying a `CartesianTwist` (by default expressed internally in `m/s` and `rad/s`) to a `CartesianPose`
153 is simply multiplying it by a time period:
154 
155 ```cpp
156 using namespace std::chrono_literals;
157 auto period = 10s;
158 
159 state_representation::CartesianTwist wVa("a", Eigen::Vector3d(1, 0, 0));
160 state_representation::CartesianPose wPa = period * wVa; // note that wVa * period is also implemented
161 ```

163 ### Cartesian state distance and norms
164 
165 As a `CartesianState` represents a spatial transformation, distance between states and norms computations have been
166 implemented. The distance functions is represented as the sum of the distance over all the state variables:
167 
168 ```cpp
169 using namespace state_representation;
170 CartesianState cs1 = CartesianState::Random("test");
171 CartesianState cs2 = CartesianState::Random("test");
172 
173 double d = cs1.dist(cs2);
174 // alternatively one can use the friend type notation
175 d = dist(cs1, cs2)
176 ```
177 
178 By default, the distance is computed over all the state variables.
179 It is worth noting that it has no physical units, but is still relevant to check how far two states are in all their features.
180 One can specify the state variable to consider using the `CartesianStateVariable` enumeration:
181 
182 ```cpp
183 // only the distance in position
184 double d_pos = cs1.dist(cs2, CartesianStateVariable::POSITION);
185 // distance over the wrench
186 double d_wrench = cs1.dist(cs2, CartesianStateVariable::WRENCH);
187 ```
188 
189 When doing so, the unit of the distance is the one of the corresponding state variable.
190 The norm of a state is using a similar API and returns the norms of each state variable:
191 
192 ```cpp
193 using namespace state_representation;
194 CartesianState cs = CartesianState::Random("test");
195 // default is norm over all the state variables, hence vector is of size 8
196 std::vector<double> norms = cs.norms();
197 // for only norm over the pose you need to specify it
198 std::vector<double> pose_norms = cs.norms(CartesianStateVariable::POSE);
199 ```
200 
201 `normalize`, inplace normalization, and the copy normalization, `normalized`, are also implemented:
202 
203 ```cpp
204 using namespace state_representation;
205 CartesianState cs = CartesianState::Random("test");
206 // inplace, default is normalization over all the state variables
207 cs.normalize()
208 // copied normalized state, with only linear velocity normalized
209 CartesianState csn = cs.normalized(CartesianStateVariable::LINEAR_VELOCITY)
210 ```

212 ## Joint state
213 
214 `JointState` follows the same logic as `CartesianState` but for representing robot states.
215 Similarly to the `CartesianState` the class `JointState`, `JointPositions`, `JointVelocities` and `JointTorques` have been developed.
216 The API follows exactly the same logic with similar operations implemented.
217 
218 A `JointState` is defined by the name of the corresponding robot and the name of each joints.
219 
220 ```cpp
221 // create a state for myrobot with 3 joints
222 state_representation::JointState js("myrobot", std::vector<string>({"joint0", "joint1", "joint2"}));
223 ```
224 
225 Note that if the joints of the robot are named `{"joint0", "joint1", ..., "jointN"}` as above,
226 you can also use the constructor that takes the number of joints as input which will name them accordingly:
227 
228 ```cpp
229 // create a state for myrobot with 3 joints named {"joint0", "joint1", "joint3"}
230 state_representation::JointState js("myrobot", 3);
231 ```
232 
233 All the getters and setters for the `positions`, `velocities`, `accelerations` and `torques` are defined for both
234 `Eigen::VectorXd` and `std::vector<double>`:
235 
236 ```cpp
237 js.set_positions(Eigen::Vector3d(.5, 1., 0.));
238 js.set_positions(std::vector<double>{.5, 1., 0.});
239 ```
240 
241 Note that when using those setters, the size of the input vector should correspond to the number of joints of the state:
242 
243 ```cpp
244 js.set_positions(Eigen::Vector4d::Random()); // will throw an IncompatibleSizeException
245 ```

247 ### Joint state operations
248 
249 Basic operations such as addition, subtraction and scaling have been implemented:
250 
251 ```cpp
252 state_representation::JointState js1("myrobot", 3);
253 state_representation::JointState js2("myrobot", 3);
254 double lambda = 0.5;
255 
256 // for those operation to be valid both js1 and js2
257 // should correspond to the same robot and have the
258 // same number of joints
259 state_representation::JointState jssum = js1 + js2;
260 state_representation::JointState jsdiff = js1 - js2;
261 state_representation::JointState jsscaled = lambda * js1;
262 ```
263 
264 Multiplication of joint states doesn't have a physical meaning and is, therefore, not implemented.
265 
266 ### Conversion between joint state variables
267 
268 Similarly to `CartesianState`, the conversion between `JointPositions` and `JointVelocities`
269 happens through operations with `std::chrono_literals`.
270 
271 ```cpp
272 using namespace std::chrono_literals;
273 auto period = 1h;
274 
275 // create a state for myrobot with 3 joints named {"joint0", "joint1", "joint3"}
276 // and provide the position values
277 state_representation::JointPositions jp("myrobot", Eigen::Vector3d(1, 0, 0));
278 
279 // result are velocities of 1 rad/h for joint0 expressed in rad/s
280 state_representation::JointVelocities jv = jp / period;
281 ```
282 
283 ```cpp
284 using namespace std::chrono_literals;
285 auto period = 10s;
286 
287 // create a state for myrobot with 3 joints named {"joint0", "joint1", "joint3"}
288 // and provide the velocities values
289 state_representation::JointVelocities wVa("a", Eigen::Vector3d(1, 0, 0));
290 
291 state_representation::JointPositions jp = period * jv; // note that jv * period is also implemented
292 ```

294 ## The Jacobian matrix
295 
296 The `Jacobian` matrix of a robot ensures the conversion between both `CartesianState` and `JointState`.
297 Similarly to the `JointState`, a `Jacobian` is associated to a robot and defined by the robot and the number of joints.
298 As it is a mapping between joint and task spaces, as for the `CartesianState`, it is also defined by an associated
299 frame name and a reference frame.
300 
301 ```cpp
302 // create a Jacobian for myrobot with 3 joints, associated to frame A and expressed in B
303 state_representation::Jacobian jac("myrobot", std::vector<string>({"joint0", "joint1", "joint2"}), "A", "B");
304 ```
305 
306 The API is the same as the `JointState`, hence the constructor can also accept the number of joints to initialize the
307 joint names vector.
308 
309 ```cpp
310 // create a Jacobian for myrobot with 3 joints named {"joint0", "joint1", "joint3"}, associated to frame A and
311 // expressed in world (default value of the reference frame when not provided)
312 state_representation::Jacobian jac("myrobot", 3, "A");
313 ```
314 
315 The `Jacobian` is simply a `6 x N` matrix where `N` is the number of joints.
316 Therefore, the data can be set from an `Eigen::MatrixXd` of correct dimensions.
317 
318 ```cpp
319 jac.set_data(Eigen::MatrixXd::Random(6, 3)); // throw an IncompatibleSizeException if the size is not correct
320 ```
321 
322 All the functionalities of the `Jacobian` have been implemented such as `transpose`, `inverse` or `pseudoinverse` functions.
323 
324 ```cpp
325 /// returns the 3 x 6 transposed matrix
326 state_representation::Jacobian jacT = jac.transpose();
327 // will throw an error as a 6 x 3 matrix is not invertible
328 state_representation::Jacobian jacInv = jac.inverse();
329 // compute the pseudoinverse without the need of being invertible
330 state_representation::Jacobian jacPinv = jac.pseudoinverse();
331 ```
332 
333 Those operations are very useful to convert `JointState` from `CartesianState` and vice versa.
334 
335 ### Conversion between JointVelocities and CartesianTwist
336 
337 The simplest conversion is to transform a `JointVelocities` into a `CartesiantTwist` by multiplication with the `Jacobian`
338 
339 ```cpp
340 state_representation::Jacobian jac("myrobot", 3, "eef_frame", "base_frame");
341 state_representation::JointVelocities jv("myrobot", 3);
342 // compute the twist of eef_frame in base_frame from the joint velocities
343 state_representation::CartesianTwist eef_twist = jac * jv;
344 ```
345 
346 The opposite transformation, from `CartesianTwist` to `JointVelocities` requires the multiplication with the `inverse`
347 (or `pseudoinverse`).
348 
349 ```cpp
350 state_representation::Jacobian jac("myrobot", 3, "eef");
351 state_representation::CartesianTwist eef_twist("eef")
352 state_representation::JointVelocities jv = jac.pseudoinverse() * eef_twist;
353 // in case of non matching frame or reference frame throw an IncompatibleStatesException
354 state_representation::CartesianTwist link2_twist("link2", "link0")
355 state_representation::JointVelocities jv = jac.pseudoinverse() * link2_twist;
356 ```
357 
358 Note that the `inverse` or `pseudoinverse` functions are computationally expensive and the `solve` function that relies
359 on the solving of the system `Ax = b` using `Eigen` has been implemented.
360 
361 ```cpp
362 state_representation::CartesianTwist eef_twist("eef")
363 // faster than doing jac.pseudoinverse() * eef_twist
364 state_representation::JointVelocities jv = jac.solve(eef_twist);
365 ```
366 
367 ### Conversion between JointTorques and CartesianWrench
368 
369 The other conversion that is implemented is the transformation from `CartesianWrench` to `JointTorques`, this one using
370 the `transpose`.
371 
372 ```cpp
373 state_representation::CartesianWrench eef_wrench("eef")
374 // faster than doing jac.pseudoinverse() * eef_twist
375 state_representation::JointTorques jt = jac.transpose() * eef_wrench;
376 ```

378 ### Matrix multiplication
379 
380 The `Jacobian` object contains an underlying `Eigen::MatrixXd` which can be retrieved using the `Jacobian::data()` method.
381 Direct multiplication of the `Jacobian` object with another `Eigen::MatrixXd` has also been implemented and returns the
382 `Eigen::MatrixXd` product.
383 
384 ```cpp
385 state_representation::Jacobian jac("myrobot", 3, "eef", Eigen::MatrixXd::Random(6, 3));
386 // alternatively can use the Random static constructor
387 state_representation::Jacobian jac = state_representation::Jacobian::Random("myrobot", 3, "eef");
388 Eigen::MatrixXd mat = jac.data();
389 Eigen::MatrixXd res = jac * Eigen::MatrixXd(3, 4); // equivalent to jac.data() * Eigen::MatrixXd(3, 4);
390 ```

392 ### Changing the Jacobian reference frame
393 
394 As stated earlier, the `Jacobian` is expressed in a reference frame and can, therefore, use the operations with
395 `CartesianPose` to be modified. It relies on the usage of the overloaded `operator*` or the `set_reference_frame`
396 function for inplace modifications. It is equivalent to multiply each columns of the `Jacobian` by the `CartesianPose`
397 on both the linear and angular part of the matrix. For this operation to be valid, the `CartesianPose` name has to
398 match the current `Jacobian` reference frame.
399 
400 ```cpp
401 state_representation::Jacobian jac = state_representation::Jacobian::Random("myrobot", 3, "eef", "base_frame");
402 state_representation::CartesianPose pose = state_representation::CartesianPose::Random("base_frame", "world");
403 // the result is the Jacobian expressed in world
404 state_representation::Jacobian jac_in_world = pose * jac;
405 // alternatively, one case use the set_reference_frame function for inplace modifications
406 jac.set_reference_frame(pose);
407 // in case of non matching operation throw an IncompatibleStatesExceptions
408 jac.set_reference_frame(state_representation::CartesianPose::Random("link0", "world"));
409 ```
