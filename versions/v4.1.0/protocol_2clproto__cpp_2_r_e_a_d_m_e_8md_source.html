1 # clproto (C++)
2 
3 This library provides a simple interface for encoding and decoding
4 control library type objects to and from a serialized binary string
5 representation (the wire format).
6 
7 ```c++
8 #include <clproto.h>
9 #include <state_representation/space/cartesian/CartesianState.hpp>
10 
11 auto state = state_representation::CartesianState::Random("frame");
12 
13 // Encode the message into the wire format
14 std::string message = clproto::encode(state);
15 
16 // The type is automatically derived from the object,
17 // but can also be provided explicitly
18 clproto::encode<state_representation::CartesianState>(state);
19 
20 // Decode the message back into an object
21 auto new_state = clproto::decode<state_representation::CartesianState>(message);
22 
23 // The previous method will throw an exception if the
24 // message cannot be parsed into that object type.
25 // For exception-safe decoding, pass an object by reference
26 // to the decode function and check the bool return value
27 state_representation::CartesianState state_reference;
28 if (clproto::decode(message, state_reference)) {
29  // successful decoding, state_reference has been modified
30 } else {
31  // unsuccessful decoding, exception is suppressed and state_reference is unmodified
32 }
33 
34 // If the message type is not known, use the following
35 // methods to check the validity and type
36 if (!clproto::is_valid(message)) {
37  // message cannot be decoded into any known type
38 }
39 
40 clproto::MessageType type = clproto::check_message_type(msg);
41 ```

43 ## Combining multiple message into a packet
44 
45 The serialized binary string encoding of a state message is not self-delimiting.
46 The consequence of this is that when multiple encoded messages are combined on the wire,
47 it is not possible to know where the first message ends and the second one starts.
48 
49 There will often be cases where it makes sense to combine multiple state messages
50 into one high level message packet. For example, a robot might broadcast the `CartesianState`
51 of its end-effector together with its `JointState`. A client controller might want to listen
52 for the combined "Robot State" on a single subscription topic.
53 
54 The clproto library facilitates this usage by providing the methods `pack_fields` and `unpack_fields`.
55 An ordered vector of encoded messages are combined into a data packet by the `pack_fields` method.
56 For N fields, it adds a message header with (N + 1) values, describing the number of fields
57 and the data size of each ordered field method. This allows the `unpack_fields` method to split
58 the encoded fields back into an ordered vector to be decoded.
59 
60 ```c++
61 #include <clproto.h>
62 #include <state_representation/space/cartesian/CartesianState.hpp>
63 #include <state_representation/robot/JointState.hpp>
64 
65 using namespace state_representation;
66 
67 // A robot can produce multiple state messages
68 auto cart_state = CartesianState::Random("robot_ee", "robot_base");
69 auto joint_state = JointState::Random("robot", 7);
70 
71 // Encode each state variable and add them to an ordered vector
72 std::vector<std::string> encoded_robot_state;
73 encoded_robot_state.emplace_back(clproto::encode(cart_state));
74 encoded_robot_state.emplace_back(clproto::encode(joint_state));
75 
76 // Pack the message fields into a raw data buffer,
77 // reserving sufficient space to contain all messages
78 char encoded_packet_buffer[2 * CLPROTO_PACKING_MAX_FIELD_LENGTH];
79 clproto::pack_fields(encoded_robot_state, encoded_packet_buffer);
80 
81 // The message fields can also be packed into a std::string,
82 // provided that sufficient space is reserved for all messages
83 std::string encoded_packet_str;
84 encoded_packet_str.reserve(2 * CLPROTO_PACKING_MAX_FIELD_LENGTH);
85 clproto::pack_fields(encoded_robot_state, encoded_packet_str.data());
86 
87 // Unpack a combined message packet back into an ordered vector of encoded state variables
88 std::vector<std::string> unpacked_encoded_robot_state = clproto::unpack_fields(encoded_packet_buffer);
89 // (or, for a string type encoded packet:)
90 unpacked_encoded_robot_state = clproto::unpack_fields(encoded_packet_str.c_str());
91 
92 // Finally, decode the encoded fields in the same order as the original packing
93 auto decoded_cart_state = clproto::decode<CartesianState>(unpacked_encoded_robot_state.at(0));
94 auto decoded_joint_state = clproto::decode<JointState>(unpacked_encoded_robot_state.at(1));
95 ```
96 
97 The pack / unpack methods are provided as a convenience only for simple synthesis of messages
98 from core state message types. The communication implementation can always be extended further
99 by the end user as necessary, for example to prepend an indication of the field types to the message,
100 or to add other delimiting behaviour.

102 ### ZMQ example
103 
104 The packing and unpacking of the message should be handled according to the network requirements.
105 This sections shows an example for ZMQ messaging.
106 
107 ```c++
108 #include <zmq.h>
109 
110 // Combine encoded state messages into an ordered vector
111 std::vector<std::string> encoded_robot_state = ...;
112 
113 // Pack and publish the combined state messages, assuming a pre-configured publishing socket
114 zmq::socket_t publisher;
115 zmq::message_t zmq_message(encoded_robot_state.size() * CLPROTO_PACKING_MAX_FIELD_LENGTH);
116 pack_fields(encoded_robot_state, static_cast<char *>(zmq_message.data()));
117 publisher.send(zmq_message, zmq::send_flags::none);
118 
119 // On the receiving side, cast the message data to a char pointer before unpacking
120 zmq::message_t received_zmq_message;
121 zmq::socket_t subscriber;
122 auto result = subscriber.recv(received_zmq_message);
123 if (result) {
124  received_encoded_robot_state = unpack_fields(static_cast<const char*>(received_zmq_message.data()));
125 }
126 ```

128 ### Reserving packet size
129 Note that, because the packing function writes to a buffer through a raw pointer, the associated
130 buffer must have enough reserved size to contain the entire packet. For simplicity, the
131 message size can be reserved as `number_of_fields * CLPROTO_PACKING_MAX_FIELD_LENGTH`, but
132 this is almost much larger than necessary. Since the resultant packed data is not self-delimiting,
133 the actual data length cannot be calculated post-hoc.
134 
135 If bandwidth is limited, the following formula can be used to calculate the minimum needed buffer size.
136 
137 ```c++
138 std::vector<std::string> encoded_fields = ...;
139 
140 // The packet header starts with N + 1 values of size clproto::field_length_t,
141 // where N is the number of fields N. The first value stores the number of fields,
142 // while the following N values store the data length of each respective field.
143 std::size_t packet_size = sizeof(clproto::field_length_t) * (encoded_fields.size() + 1);
144 
145 // Add the size of each field
146 for (const auto& field : encoded_fields) {
147  packet_size += field.size();
148 }
149 ```
